# 陪玩平台 (PlayBuddy) 完整项目设计文档

## 一、项目概述

基于 GBaseAdmin 框架扩展的陪玩平台，使用 `play` 作为应用前缀，所有表名格式为 `play_{module}`。后端为 GoFrame v2 MonoRepo 架构，前端为 Vben Admin (web-antd)。通过代码生成器 `codegen` 从数据库表直接生成 90% 以上的 CRUD 代码，仅需手动编写订单状态流转、支付回调、利润计算等核心业务逻辑。

## 二、模块划分

| 模块组 | 表名前缀 | 说明 | 管理端/C端 |
|--------|----------|------|-----------|
| 会员模块 | play_member, play_member_level | 会员 + 等级体系 | 后台管理 + C端 |
| 陪玩师模块 | play_coach, play_coach_level, play_coach_apply | 陪玩师 + 等级 + 申请审核 | 后台管理 + C端 |
| 店铺模块 | play_shop | 陪玩师绑定店铺 | 后台管理 |
| 商品模块 | play_category, play_goods | 分类（树形）+ 商品 | 后台管理 |
| 订单模块 | play_order, play_order_item | 订单 + 订单明细 | 后台管理 + C端 |
| 支付模块 | play_payment | 支付记录 | 后台管理 |
| 充值模块 | play_recharge_plan, play_recharge_order, play_balance_log | 充值项 + 充值订单 + 余额流水 | 后台管理 + C端 |
| 活动模块 | play_activity, play_activity_reward, play_activity_step, play_activity_join | 活动 + 奖励配置 + 步骤引导 + 参与记录 | 后台管理 + C端 |
| 优惠券模块 | play_coupon, play_coupon_member | 优惠券（含新人券） + 领取记录 | 后台管理 + C端 |
| 第三方登录 | play_oauth | 第三方账号绑定 | C端 |
| 评价模块 | play_review | 订单评价 | 后台管理 + C端 |
| 利润统计 | play_profit_log | 利润分成流水 | 后台管理 |

## 四、字段与代码生成器映射说明

以下说明每个表的字段如何被 codegen 自动识别：

| 字段模式 | 触发规则 | 生成结果 |
|----------|----------|----------|
| `parent_id` (play_category) | 自动识别树形 | Tree API + 前端树形表格 + TreeSelect 组件 |
| `*_id` (member_id, coach_id, shop_id, category_id, coupon_id, order_id) | 外键关联 | Select 下拉 + 关联查询显示字段 |
| `*_image` (cover_image, logo_image, id_card_front_image, review_image) | 图片上传 | ImageUpload 组件 |
| `avatar` | 头像 | ImageUpload 组件 |
| `*_content` (desc_content, review_content, reply_content, callback_content) | 富文本 | RichText 编辑器 |
| `password` | 密码 | bcrypt 加密 + PasswordInput |
| `icon` | 图标选择 | IconPicker 组件 |
| `sort` | 排序 | InputNumber 组件 |
| `*_at` (pay_at, start_at, finish_at, cancel_at, expire_at...) | 时间 | DateTimePicker 组件 |
| `*_url` (link_url) | URL输入 | InputUrl 组件 |
| `status` (2值) | Switch | Switch 开关组件 |
| `type/level` + 枚举注释 | 枚举 | Select 下拉组件 + consts 常量 |
| `is_*` (is_coach, is_online, is_anonymous) 2值 | Switch | Switch 开关组件 |
| `audit_status` 3值枚举 | Radio | Radio 单选组件 |
| `pay_type`, `pay_status`, `order_status`, `biz_type`, `use_status`, `provider`, `gender` | 枚举 | Select/Radio + consts + Tag 颜色 |
| `skill_desc` (TEXT) | 大文本 | Textarea 组件 |
| 公共字段 (created_at, updated_at, deleted_at, created_by, dept_id) | 自动跳过 | 不生成表单字段 |

## 七、核心业务逻辑设计

### 7.1 订单状态流转

```
                  用户下单
                    │
                    ▼
              ┌─────────┐
              │ 0=待支付  │◄──────────────────────┐
              └────┬────┘                        │
                   │ 支付成功                     │ 支付超时自动取消(30分钟)
                   ▼                              │
              ┌─────────┐                    ┌────┴────┐
              │ 1=已支付  │────────────────►  │ 4=已取消  │
              └────┬────┘  用户取消(未接单前)  └─────────┘
                   │ 陪玩师接单
                   ▼
              ┌─────────┐
              │ 2=进行中  │────────────────►  ┌─────────┐
              └────┬────┘  用户申请退款       │ 5=退款中  │
                   │ 陪玩师完成服务            └────┬────┘
                   ▼                               │ 退款处理完成
              ┌─────────┐                    ┌────▼────┐
              │ 3=已完成  │                   │ 6=已退款  │
              └─────────┘                    └─────────┘
                   │
                   ▼ 用户评价(可选)
              写入 play_review
```

**状态流转规则**：
- `0->1`：支付成功回调触发，记录 pay_at
- `0->4`：30 分钟未支付自动取消，或用户主动取消
- `1->2`：陪玩师接单，记录 start_at
- `1->4`：用户在陪玩师接单前可取消，触发原路退款
- `2->3`：陪玩师完成服务，记录 finish_at，触发利润分成
- `2->5`：用户申请退款，进入退款审核流程
- `5->6`：退款处理完成，记录退款金额
- `3->评价`：订单完成后用户可评价

### 7.2 利润分成逻辑

订单完成时（`order_status` 从 2 变为 3），触发利润分成计算：

```
实付金额 (pay_amount)
    │
    ├── 平台抽成 = pay_amount × platform_rate%
    │   （platform_rate 来自陪玩师等级表 play_coach_level.commission_rate）
    │
    ├── 店铺抽成 = (pay_amount - 平台抽成) × shop_rate%
    │   （shop_rate 来自店铺表 play_shop.commission_rate，无店铺则为 0）
    │
    └── 陪玩师收入 = pay_amount - 平台抽成 - 店铺抽成
```

**示例**：
- 订单实付 100 元（10000 分）
- 陪玩师等级为"黄金"，平台抽成 20%
- 陪玩师绑定店铺，店铺抽成 10%
- 平台抽成 = 10000 x 20% = 2000 分
- 店铺抽成 = (10000 - 2000) x 10% = 800 分
- 陪玩师收入 = 10000 - 2000 - 800 = 7200 分

分成结果写入 `play_profit_log` 表，同时更新 `play_coach.income_total` 和 `play_coach.income_balance`。

### 7.3 优惠券使用逻辑

**领取流程**：
1. 检查优惠券状态 (`status=1`) 且在有效期内
2. 检查是否超过发放总量 (`claim_num < total_num` 或 `total_num=0`)
3. 检查该会员领取数量是否超过 `per_limit`
4. **新人券校验**：若 `is_new_member=1`，检查会员注册时间 `play_member.created_at`，距今是否在 `new_member_days` 天内，超过则拒绝领取
5. 写入 `play_coupon_member` 记录，`use_status=0`，设置 `expire_at`
6. `play_coupon.claim_num` 加 1

**下单使用流程**：
1. 查询会员可用优惠券（`use_status=0` 且 `expire_at > NOW()`）
2. 验证订单金额是否满足 `min_amount`
3. 计算优惠金额：
   - 满减券/无门槛券：直接抵扣 `face_value`，不超过订单金额
   - 折扣券：`total_amount × (100 - face_value) / 100`，即折扣金额
4. 更新 `play_coupon_member.use_status=1`，记录 `use_at` 和 `order_id`
5. 更新 `play_coupon.used_num` 加 1

**过期处理**（定时任务）：
- 每天凌晨扫描 `play_coupon_member`，将 `use_status=0` 且 `expire_at < NOW()` 的记录更新为 `use_status=2`

### 7.4 充值与余额流水

**充值流程**：
1. 用户选择充值方案，创建 `play_recharge_order`
2. 调用微信/支付宝支付
3. 支付回调成功后：
   - 更新 `play_recharge_order.pay_status=1`
   - 增加会员余额：`play_member.balance += amount + gift_amount`
   - 写入余额流水 `play_balance_log`（`biz_type=1`）

**消费流程**（余额支付）：
1. 校验余额是否充足
2. 扣减 `play_member.balance`
3. 写入余额流水（`biz_type=2`，`change_amount` 为负数）
4. 更新订单支付状态

**退款流程**：
1. 增加 `play_member.balance`
2. 写入余额流水（`biz_type=3`，`change_amount` 为正数）

### 7.5 双身份切换

JWT claims 中增加 `current_role` 字段：

```go
type Claims struct {
    UserID      int64  `json:"userId"`
    Username    string `json:"username"`
    DeptID      int64  `json:"deptId"`
    MemberID    int64  `json:"memberId"`     // 会员ID
    IsCoach     int    `json:"isCoach"`       // 是否是陪玩师
    CoachID     int64  `json:"coachId"`       // 陪玩师ID（is_coach=1 时有值）
    CurrentRole string `json:"currentRole"`   // 当前身份：member / coach
}
```

- `/member/switch_role` 接口重新签发 JWT，切换 `currentRole`
- C端中间件根据 `currentRole` 控制陪玩师专属接口访问权限

| 下单活动 | 2 | 下单满额/满次 | 如下单满3次送优惠券 |
| 注册活动 | 3 | 新用户注册 | 注册即送奖励 |
| 图文步骤活动 | 4 | 完成所有步骤 | 用户按步骤引导操作，全部完成后领奖 |
| 自定义活动 | 5 | 无条件/需报名 | 运营灵活配置 |

**多种奖励机制**：
- 一个活动可配置多条 `play_activity_reward` 记录，支持同时发放多种奖励
- 奖励类型：余额（reward_type=1）、优惠券（reward_type=2）、经验值（reward_type=3）、会员等级天数（reward_type=4）
- 用户完成活动后，系统遍历该活动所有奖励记录，逐一发放

**活动参与流程**：

```
用户浏览活动列表
    │
    ▼
查看活动详情（含步骤列表 + 奖励列表）
    │
    ▼
┌─────────────────────────────────────────────┐
│ condition_type 判断                          │
├─────────────────────────────────────────────┤
│ 0=无条件  → 直接参与                         │
│ 1=需报名  → 用户点击报名，创建 join 记录      │
│ 2=充值满额 → 系统在充值回调中自动检测          │
│ 3=下单满额 → 系统在订单完成时自动检测          │
│ 4=完成步骤 → 用户逐步完成图文步骤             │
└─────────────────────────────────────────────┘
    │
    ▼ 满足条件
用户点击"完成活动"
    │
    ▼
┌─────────────────────────────────────────────┐
│ is_auto_reward 判断                          │
├─────────────────────────────────────────────┤
│ 1=自动发奖 → 立即遍历 activity_reward 发放    │
│ 0=需审核   → join_status 设为2(已完成)，      │
│              等待后台审核后手动发奖             │
└─────────────────────────────────────────────┘
    │
    ▼ 发放奖励
遍历 play_activity_reward：
  ├── reward_type=1(余额) → play_member.balance 增加，写 balance_log(biz_type=4)
  ├── reward_type=2(优惠券) → 自动写入 play_coupon_member 记录
  ├── reward_type=3(经验值) → play_member.exp 增加，检查是否升级
  └── reward_type=4(等级天数) → 延长会员等级有效期
    │
    ▼
更新 join_status=3(已领奖)，记录 reward_at
```

**图文步骤活动流程**：
1. 后台创建活动时，配置 `play_activity_step` 步骤列表（含标题、说明、示例图片）
2. C端展示步骤列表，用户按顺序查看每个步骤的图文引导
3. 用户完成当前步骤后，调用 `/activity/complete_step` 接口，`current_step` 加 1
4. 当 `current_step` 等于总步骤数时，`join_status` 变为可完成状态
5. 用户点击"完成活动"，触发奖励发放

**充值/下单活动自动检测**：
- 充值回调成功后，查询当前生效的充值活动（type=1），检查用户累计充值是否满足 `condition_value`
- 订单完成后，查询当前生效的下单活动（type=2），检查用户累计下单次数/金额是否满足 `condition_value`
- 满足条件时自动创建 `play_activity_join` 记录并发放奖励

## 十一、实施计划

### 阶段一：基础设施（第 1~2 天）

4. 在 `app/play/internal/cmd/cmd.go` 中注册全部路由

6. 编译运行验证全部 CRUD 接口可用
7. 前端页面联调验证

**codegen 会自动生成的内容**：
- 19 个模块的 API 结构体 (api.tpl)
- 19 个模块的 Controller (controller.tpl)
- 19 个模块的 Logic/CRUD 实现 (logic.tpl)
- 19 个模块的 Service 接口 (service.tpl)
- 19 个模块的 Model DTO (model.tpl)
- 枚举常量文件 (consts.tpl)
- 19 个模块的前端 API + 类型 + 列表页 + 表单页

### 阶段二：核心业务逻辑（第 3~5 天）

**目标**：手动实现订单流转、支付、利润分成

1. **订单逻辑** (`logic/order/`):
   - 创建订单：计算会员折扣 + 优惠券抵扣
   - 状态流转方法：Pay / Accept / Start / Finish / Cancel / Refund
   - 订单完成时触发利润分成写入 `play_profit_log`

2. **支付逻辑** (`logic/payment/`):
   - 集成 `utility/wechat/pay.go` 和 `utility/alipay/pay.go`
   - 支付回调处理：验签 -> 更新支付记录 -> 更新订单状态
   - 余额支付：扣减余额 + 余额流水

3. **利润分成逻辑** (`logic/profit_log/`):
   - 查询陪玩师等级获取 platform_rate
   - 查询店铺获取 shop_rate（无店铺则为 0）
   - 计算三方分成并写入流水
   - 更新陪玩师收入余额

4. **充值逻辑** (`logic/recharge_order/`):
   - 创建充值订单 -> 支付 -> 回调后增加余额 + 写流水

| 登录 | 手机号 + 微信公众号授权 | 微信一键登录 (wx.login) | 抖音一键登录 | 手机号 + 第三方 |
| 支付 | 微信 JSAPI / 支付宝 H5 | 微信小程序支付 | 抖音支付 | 微信/支付宝 SDK |
| 分享 | 微信 JS-SDK | 小程序原生分享 | 抖音原生分享 | 系统分享 |
| 消息推送 | 公众号模板消息 | 小程序订阅消息 | 抖音订阅消息 | App 推送 |

## 十三、关键设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 金额存储 | BIGINT（分） | 避免浮点精度问题，前端展示时除以 100 |
| 评分存储 | INT（乘100） | 4.5分存为450，避免浮点 |
| 折扣/比例存储 | INT（百分比） | 85折存为85，20%存为20 |
| 后台 vs C端 | 统一 play 应用，playapi 放在 play 内 | _api.go 命名区分，路由前缀 /api/playapi，减少代码量，方便统一管理 |
| 商品分类 | parent_id 树形 | 直接利用 codegen 自动生成树形 API |
| 订单金额冗余 | goods_title/goods_price 冗余到订单 | 商品修改不影响历史订单 |
| 陪玩师与会员 | 一对一关联，member.is_coach 标记 | 同一账号双身份，JWT 切换 |
| 店铺抽成基数 | 扣除平台抽成后的金额 | 避免平台+店铺抽成超过100%的风险 |
| 优惠券过期 | 定时任务批量处理 | 不在查询时实时判断，减少逻辑复杂度 |
| WAP 端跨端框架 | Taro 4.x + React 18 | 一套代码覆盖 H5、微信/抖音小程序、App，React 一等公民 |
| WAP 端 UI 组件库 | NutUI-React | 京东官方出品，专为 Taro 设计，多端兼容性好 |
| WAP 端状态管理 | Zustand | 轻量无 boilerplate，React 友好 |
| WAP 端项目位置 | wap-ui 目录 | 与 admin-go、vue-vben-admin 平级，MonoRepo 统一管理 |
